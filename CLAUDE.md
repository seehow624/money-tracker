## What this is

A personal, single-user finance tracker. Self-hosted (e.g. on a small always-on machine over your LAN/VPN). Optionally drives a Telegram bot that POSTs natural-language money messages ("ramen 12", "transfer 2k checking to visa") to the JSON API.

Not a multi-tenant SaaS. One user, one SQLite file. The UI is protected by a simple username/password session (see `.env.example`).

## Commands

```bash
npm run dev              # local dev (localhost:3000)
npm run dev:lan          # bind 0.0.0.0 — needed for phone testing on LAN
npm run build && npm start
npm run typecheck        # tsc --noEmit, the only check that runs

# database
npm run db:generate      # drizzle-kit generate after editing src/db/schema.ts
npm run db:migrate       # apply migrations (scripts/migrate.ts)
npm run db:studio        # drizzle-kit studio
npm run db:seed          # seed example accounts + categories (idempotent)

# scheduled jobs (wire these into cron/launchd/systemd yourself)
npm run db:run-scheduled # run scheduled_rules due today
npm run db:daily-backup  # backup money.db, prune > 30 days
npm run db:fetch-fx      # fetch FX rates from Frankfurter
```

There are no tests and no linter — `tsc --noEmit` is the verification step. Expect `npm run typecheck` to pass before claiming a change is done.

## Database

SQLite via `better-sqlite3` + Drizzle ORM. Schema lives in `src/db/schema.ts`; the live file is `data/money.db` (gitignored). Override with `MONEY_TRACKER_DB_PATH`. WAL mode + foreign keys are enabled in `src/db/index.ts`.

Migrations are SQL files in `drizzle/`, generated by `drizzle-kit`. Edit `schema.ts`, then `npm run db:generate`, then `npm run db:migrate`.

Both `db` (Drizzle) and `rawDb` (raw better-sqlite3) are exported. Most code uses `db`; raw SQL is reserved for things Drizzle expresses awkwardly (GROUP BY ROUND, dynamic IN clauses, multi-statement updates).

## Architecture

**Single shared module of read queries.** `src/lib/queries.ts` is the source of truth for every aggregate the UI shows: monthly summaries, category breakdowns, year/cumulative trends, account balances with credit-card cycles, monthly day-grouped logs, fuzzy search. Pages call these from server components — they don't write SQL inline. When adding a new view, add or extend a function here rather than querying the DB from the page.

**Money flow & sign convention.**
- `transactions.amount` is the native amount in `currency`; `amountMyr = amount * fxRate` is the base-currency equivalent at the transaction date. Conversion happens in `src/lib/fx.ts` (`convertToMyr`), which falls back to the most recent prior rate when the exact date is missing (Frankfurter has no weekend rates).
- `transfer` rows have both `account_id` (source) and `to_account_id` (destination). Income/expense rows only use `account_id`. `categoryId` is null for transfers.
- For non-CC accounts, balance = `startingBalance + Σincome − Σexpense − Σtransfer_out + Σtransfer_in` (over rows on/after `startingBalanceDate`).
- For credit cards, the math inverts and uses `statementDay` / `paymentDay` to compute a cycle (`ccCycle()` in `queries.ts`). The displayed balance is `balancePayable + outstanding`. `startingBalance` on a CC is a manual adjustment knob — set it negative to absorb cumulative imbalance from incomplete history. Don't "fix" the sign without understanding this.

**Fuzzy resolution.** The API ingests human strings ("visa", "checking", "ewallet"). `src/lib/fuzzy.ts` has alias tables and a 4-stage match (alias → exact → starts-with → substring → token-wise). When adding accounts or categories, also add aliases here if they have common shorthands — otherwise the bot will fail to record them.

**App Router structure.** Server components render pages; mutations go through server actions in `src/app/**/actions.ts` (e.g. `transactions/actions.ts`, `balances/actions.ts`, `more/actions.ts`). Pages set `export const dynamic = 'force-dynamic'` since data changes constantly. Actions call `revalidatePath('/')`, `'/stats'`, `'/balances'` after writes — keep that pattern.

**API routes** (`src/app/api/*`) are the bot surface. All require `Authorization: Bearer ${MONEY_TRACKER_API_TOKEN}` via `checkAuth()` in `src/lib/api-auth.ts`.

**In-app AI assistant** (`src/app/api/ai/{chat,undo}`, `src/components/AIInput.tsx`) is a separate NLU path. It uses the `openai` SDK against any OpenAI-compatible endpoint and runs a tool-calling loop. You maintain a list of provider profiles in the `ai_profiles` table (Ollama, OpenAI, OpenRouter, etc.) and pick a primary + optional fallback (profile, model) pair via the `app_settings` keys `ai.{primary,fallback}{ProfileId,Model}` — both are editable at `/more/ai`. **API keys for these providers live in the `ai_profiles` table inside `data/money.db` (gitignored), never in the repo.** `src/lib/settings.ts` exposes profile CRUD plus `getAIRouting()`; the chat route calls primary first and retries once with the fallback on any error (returning `fellBack: { from, reason }` so the UI can flag which model answered). Tools and the system prompt live in `src/lib/ai-tools.ts` — adding a tool means extending both `TOOLS` and the `executeTool` switch. AI-recorded transactions get `source: 'ai'`; `/api/ai/undo` only deletes rows with that source. Account/category resolution goes through `src/lib/fuzzy.ts`.

**`transactions.source`** distinguishes entry origin: `'manual'` (UI), a `'telegram:*'` tag (Telegram API), `'ai'` (in-app assistant), and historical importer values. Preserve this when adding new entry surfaces.

**Scheduled rules** (`scheduled_rules` table) generate recurring transactions. `scripts/run-scheduled.ts` runs daily, catches up multiple periods if the cron skipped, and writes one transaction per due date. Recurring detection (`recurringDetections()` in queries.ts) is a separate read-only heuristic over historical transactions — two different systems.

**Reminders** (`reminder_settings`, `reminder_sent`) — daily entry nudge, bill due, budget warning. State is dedup'd by `(kind, date, account_id)`.

## Deployment

Self-host on any always-on machine. Run `npm run build && npm start` (or `npm run start:lan` to bind `0.0.0.0`) under a process manager — launchd, systemd, pm2, Docker, etc. Wire the three scheduled jobs (`db:run-scheduled`, `db:daily-backup`, `db:fetch-fx`) into cron or your init system. Put the app behind your own reverse proxy / VPN; the UI auth is intentionally minimal.

## Conventions specific to this codebase

- Dates are ISO `YYYY-MM-DD` strings everywhere; months are `YYYY-MM`. Avoid `Date` arithmetic on local time — use `Date.UTC()` and `.toISOString().slice(0, 10)` (see `lastStatementDate`, `nextDateOnDay` in queries.ts).
- The app is multi-currency: each account has a `currency`, and amounts are converted to a single base currency (`fmtMyr`) for cross-account totals. Conversion is real, not cosmetic.
- `paid_by` is free-text but conventionally `'me'`, `'mom'`, `'dad'`. The `paidByOthers` aggregation excludes `'me'`.
- `scripts/*.ts` referenced in `package.json` are the canonical maintenance scripts. `scripts/parse-tng.mjs` is an example statement parser — adapt it for your own statement formats.
